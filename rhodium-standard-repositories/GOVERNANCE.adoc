== Overview

The Rhodium Standard Repositories project uses the *Tri-Perimeter Contribution Framework (TPCF)* for graduated trust and distributed governance. This document defines how decisions are made, who has what authority, and how the project evolves.

=== Core Principles

* **Consensus-seeking**: We prefer agreement over voting when possible
* **Graduated trust**: Permissions grow with demonstrated reliability
* **Reversibility**: Most decisions can be revisited and changed
* **Transparency**: All governance processes are public
* **Community over ego**: Technical merit and collaboration trump seniority

=== Document Scope

This governance model applies to:

* The main `rhodium-standard-repositories` project
* All sub-projects and examples maintained under this umbrella
* Community spaces (Matrix, GitLab, future forums)
* Project resources (domains, infrastructure, social media)

== Decision-Making Process

=== Types of Decisions

We categorize decisions by impact and use appropriate processes:

[cols="1,2,2,2",options="header"]
|===
| Type | Examples | Process | Required Approval

| **Routine**
| Bug fixes, typos, clarifications
| Single maintainer approval
| 1 maintainer

| **Minor**
| New examples, documentation improvements, small features
| Maintainer consensus (no objections)
| 2 maintainers

| **Major**
| Architectural changes, new perimeters, policy changes
| Consensus with fallback to vote
| Core team consensus or 2/3 vote

| **Critical**
| License changes, CoC changes, project dissolution
| Supermajority vote + community input period
| 2/3 core team + 2-week community feedback
|===

=== Consensus Process

Our preferred decision-making method:

1. **Proposal**: Issue or merge request created with `~governance` label
2. **Discussion**: Minimum 3 business days for feedback
3. **Refinement**: Proposal updated based on input
4. **Consensus check**: Maintainers indicate support/concerns/objections
5. **Resolution**:
   * No objections = consensus achieved
   * Minor concerns = proposal proceeds with noted reservations
   * Major objections = more discussion or vote

=== Voting Process

When consensus cannot be reached:

*Eligibility*: Core maintainers only

*Quorum*: 2/3 of core maintainers must participate

*Thresholds*:
* **Simple majority** (>50%): Minor and routine decisions
* **Supermajority** (â‰¥66.67%): Major changes
* **Unanimous** (100%): Critical changes (or 2-week community feedback + 2/3)

*Timeline*:
* Voting period: 7 days minimum
* Extension: Any maintainer can request +7 days once
* Early close: Unanimous votes can close early

*Procedure*:
1. Issue labeled `~vote` created with clear proposal
2. Maintainers vote: âœ… Yes, âŒ No, ðŸ¤” Abstain
3. Abstentions don't count toward quorum
4. Result announced, rationale documented
5. Implementation timeline specified

== Project Roles

=== Community Contributor

*Requirements*: None - anyone can contribute!

*Permissions*:
* Fork repository
* Open issues
* Submit merge requests
* Participate in discussions

*Responsibilities*:
* Follow CODE_OF_CONDUCT
* Respect maintainer decisions
* Provide constructive feedback

*Recognition*: Listed in git history and .well-known/humans.txt

=== Regular Contributor

*Requirements*:
* 5+ merged contributions
* 1+ month of engagement

*Permissions*: All community contributor permissions plus:
* Issue triage help (add labels, close duplicates)
* Informal code review input

*Responsibilities*:
* Help newcomers
* Maintain quality standards in your contributions

*Recognition*: Mentioned in release notes for significant contributions

=== Trusted Contributor

*Requirements*:
* 20+ merged contributions OR demonstrated expertise
* 3+ months of consistent activity
* Positive community interactions
* Nominated by maintainer or self-nomination

*Permissions*:
* Approve MRs in your domain (after review by another trusted contributor or maintainer)
* Close issues in your domain
* Push to feature branches (not main)

*Responsibilities*:
* Maintain your area of expertise
* Review MRs promptly (within 1 week)
* Participate in architectural discussions for your domain
* Mentor community contributors

*Recognition*: Listed in MAINTAINERS.md as trusted contributor

=== Core Maintainer

*Requirements*:
* 6+ months as trusted contributor
* Significant architectural contributions
* Deep RSR philosophy understanding
* Community leadership and mentorship
* Security and governance training completed
* Nominated by core maintainer, consensus approval

*Permissions*:
* Write access to main branch
* Final decision authority in consensus process
* Voting rights on governance decisions
* Access to security issues and infrastructure

*Responsibilities*:
* Review MRs across all areas
* Make architectural decisions
* Handle security issues
* Approve new trusted contributors
* Participate in consensus decision-making
* Ensure project health and sustainability
* Mentor trusted contributors

*Recognition*: Listed in MAINTAINERS.md as core maintainer

*Time Commitment*: 5-10 hours/week expected

== Release Management

=== Versioning

We follow **Semantic Versioning 2.0.0**:

* **MAJOR** (X.0.0): Breaking changes, architectural overhauls
* **MINOR** (x.Y.0): New features, backwards-compatible
* **PATCH** (x.y.Z): Bug fixes, security patches

=== Release Cadence

* **Major releases**: When ready (no schedule)
* **Minor releases**: Quarterly (Jan, Apr, Jul, Oct)
* **Patch releases**: As needed for bugs/security

=== Release Process

1. **Preparation** (1 week before):
   * Create release branch: `release/vX.Y.Z`
   * Update version numbers
   * Move CHANGELOG `[Unreleased]` to version section
   * Run full validation: `just validate`
   * Build all examples: `just build-all`
   * Generate SBOM: `just sbom-generate`

2. **Review** (3 days):
   * Core team reviews release branch
   * Test examples and audit script
   * Verify documentation accuracy

3. **Release** (release day):
   * Merge release branch to main
   * Create signed git tag: `git tag -s vX.Y.Z -m "Release vX.Y.Z"`
   * Push tag: `git push origin vX.Y.Z`
   * Publish GitLab release with notes
   * Update badges in README.adoc if compliance changed
   * Announce on Matrix, GitLab, social media

4. **Post-Release** (1 week after):
   * Monitor for issues
   * Address critical bugs with patch release
   * Close related issues and milestone

=== Release Roles

* **Release Manager**: Core maintainer who coordinates release
* **QA Lead**: Tests examples and validates compliance
* **Documentation Lead**: Ensures docs are current and accurate
* **Communications Lead**: Handles announcements and social media

Roles rotate quarterly among core maintainers.

== Conflict Resolution

We expect disagreementsâ€”it's healthy! Here's how we handle them:

=== Process

1. **Direct Communication** (first step):
   * Talk directly with person you disagree with
   * Assume good intent
   * Focus on issues, not people
   * Use private channels if sensitive

2. **Mediation** (if direct communication doesn't resolve):
   * Request mediation from uninvolved core maintainer
   * Mediator facilitates conversation, doesn't decide
   * Focus on finding win-win solutions

3. **Escalation** (if mediation doesn't resolve):
   * Bring to full core team for decision
   * Follow consensus or voting process
   * Document decision and rationale

4. **Appeal** (if you disagree with core team decision):
   * Submit appeal within 7 days
   * Core team reconsiders with fresh perspective
   * Final decision made, documented

=== Code of Conduct Violations

CoC violations bypass normal conflict resolution:

1. Report to code-of-conduct@rhodium-standard.org
2. CoC response team investigates (2 core maintainers, not involved)
3. Response within 48 hours (acknowledgement), 1 week (resolution)
4. Actions range from warning to permanent ban
5. Appeal process available (see CODE_OF_CONDUCT)

== Succession Planning

We plan for continuity to ensure project longevity.

=== Bus Factor Mitigation

* **Minimum 3 core maintainers** at all times
* **Cross-training**: No single points of failure
* **Documentation**: All processes documented (this file!)
* **Credential rotation**: Multiple people have access to infrastructure

=== Maintainer Succession

When a core maintainer steps down:

1. Announce intention (private or public)
2. Transition period (2 weeks ideal, flexible)
3. Update MAINTAINERS.md
4. Transfer responsibilities to remaining maintainers
5. Optionally move to emeritus status

When a core maintainer is incapacitated:

1. Remaining maintainers assume responsibilities
2. Promote trusted contributor to core if needed (expedited process)
3. Attempt to contact incapacitated maintainer
4. After 6 months, formally transition to emeritus

=== Project Archival

If project becomes unmaintained:

* Post prominent notice in README
* Archive repository (read-only)
* Update security.txt with end-of-support date
* Offer to transfer to new maintainer team (requires 2/3 approval of community)

== Financial Governance

Financial decisions follow special processes due to legal and ethical implications.

=== Financial Transparency

* All funds tracked via OpenCollective (public ledger)
* Quarterly financial reports published
* Major expenditures (>$500) documented in issues

=== Budget Allocation

Percentage targets (flexible based on actual needs):

* 40%: Maintainer compensation
* 20%: Infrastructure and tooling
* 15%: Security auditing and formal verification
* 10%: Community events and outreach
* 10%: Upstream dependency support
* 5%: Emergency fund reserve

=== Spending Authority

|===
| Amount | Authority | Process

| Any core maintainer

| $100-$500

| Propose in issue, get approval, spend
